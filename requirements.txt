# 后端开发大师级提示词：Minecraft Spark AI 性能诊断平台

你现在是一名资深全栈后端架构师、Node.js/Fastify 专家、MySQL/Prisma 专家、安全工程师、Minecraft Java 服务端性能分析专家。请你从零开始实现一个生产可用的后端项目，项目名称为：

`Spark AI Analyzer Backend`

## 一、项目目标

我要开发一个面向 Minecraft 服主，尤其是小白服主的 spark 报告 AI 分析平台。

spark 是 Minecraft 服务端常用的专业性能分析模组/插件，但很多小白服主拿到 spark 分析网页后，看不懂 TPS、MSPT、主线程、火焰图、插件/模组来源、GC、内存、线程调用等数据。

本项目后端的目标是：

用户只需要粘贴一个 spark 官方 viewer 链接，后端自动抓取 spark 报告数据，整理成结构化摘要，然后调用 DeepSeek API，用专业提示词生成一份中文诊断报告，最后返回给前端展示。

## 五、spark 链接解析要求

只允许以下形式的链接：

```text
https://spark.lucko.me/{code}
https://spark.lucko.me/{code}?xxx
```

其中 `{code}` 是 spark 报告 ID。

必须做以下校验：

1. 协议必须是 HTTPS。
2. hostname 必须严格等于 `spark.lucko.me`。
3. 禁止任何其他域名。
4. 禁止用户自定义端口。
5. 禁止用户名密码形式 URL。
6. 禁止 URL fragment 影响解析。
7. 禁止 `@` 绕过。
8. 禁止跳转到其他域名。
9. 禁止请求内网 IP、localhost、metadata IP。
10. 禁止任何 file、ftp、gopher 等协议。
11. 最终请求 URL 必须由后端自己构造，不直接使用用户原始 URL。

正确方式：

```text
用户输入 URL
→ parse URL
→ 校验 hostname、protocol、pathname
→ 提取 code
→ 后端重新拼接：
   https://spark.lucko.me/{code}?raw=1
```

如果需要 full data，则由后端拼接：

```text
https://spark.lucko.me/{code}?raw=1&full=true
```

不要直接请求用户传入的完整 URL。

## 七、规则预分析模块

请实现 `SparkRuleAnalyzer`。

AI 不是唯一判断来源。后端必须先做一层规则预分析，生成结构化证据。

规则示例：

### 7.1 TPS/MSPT

Minecraft 服务端目标是 20 TPS，每 tick 理论预算约 50ms。

规则：

```text
平均 MSPT < 40：健康
40 <= 平均 MSPT < 50：接近压力边界
平均 MSPT >= 50：明显卡顿风险
最大 MSPT 明显高于平均值：可能存在偶发卡顿
TPS 长期低于 19.5：存在性能问题
```

### 7.2 主线程

如果热点集中在 Server thread / main thread：

```text
判定为主线程瓶颈
优先分析实体、区块、插件同步任务、世界保存、红石、寻路、数据库同步 IO
```

### 7.3 异步线程

如果热点集中在 async/worker 线程：

```text
不直接等价于 TPS 卡顿
需要判断是否反向阻塞主线程
可能是数据库、网络、地图渲染、Web 请求、异步保存
```

### 7.4 wait/sleep

如果出现 waitForNextTick、sleep、park、LockSupport 等：

```text
不能简单判定为问题
高 sleep 往往表示服务器有空闲时间
低 sleep 且主线程热点明显才更危险
```

### 7.5 GC/内存

如果内存使用长期接近上限，或 GC 频繁：

```text
判定为内存压力或 GC 压力
建议检查 JVM 参数、内存分配、内存泄漏、大型缓存、实体/区块堆积
```

### 7.6 常见热点关键词

识别以下关键词并生成疑似原因：

```text
chunk, region, ticket, load, unload
entity, mob, goal, pathfind, brain, ai
redstone, block update, neighbor update
hopper, inventory
database, mysql, sqlite, hikari
luckperms, essentials, dynmap, squaremap, bluemap
world save, autosave
network, packet
gc, garbage, allocation
```

输出结构：

```ts
{
  severity: "normal" | "low" | "medium" | "high" | "critical"
  summary: string
  evidence: Array<{
    title: string
    detail: string
    confidence: "high" | "medium" | "low"
  }>
  suspectedCauses: Array<{
    name: string
    category: "plugin" | "mod" | "world" | "entity" | "chunk" | "redstone" | "memory" | "jvm" | "database" | "unknown"
    reason: string
    priority: number
    confidence: "high" | "medium" | "low"
  }>
  recommendedCommands: string[]
  limitations: string[]
}
```
