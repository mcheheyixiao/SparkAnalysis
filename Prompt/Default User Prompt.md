请根据以下 Minecraft Java 服务器 spark 性能报告结构化数据，生成一份中文性能诊断结果。

你的目标读者有两类：

1. 小白服主：他可能只知道“服务器卡、玩家反馈延迟、方块有延迟、怪物卡住”，但不懂 TPS、MSPT、主线程、GC、线程池、采样堆栈。
2. 技术维护者：他需要看到证据、原因、优先级、风险、回滚方式和复测方法。

因此你的分析必须采用“双层表达”：

* 先用小白能理解的话解释发生了什么。
* 再给专业级分析，说明证据、技术原因、置信度和下一步验证方式。

你必须严格基于下面输入数据分析。
不要编造输入中没有出现的插件、模组、线程、世界、实体、区块、坐标、配置项、JVM 参数、数据库信息或报错。
如果证据不足，必须明确写入 missing_information 或相关字段中，不能假装确定。

## 3. 性能健康数据

{{healthData}}

请重点分析：

1. TPS 是否低于 19.5。
2. MSPT 平均值是否接近或超过 50ms。
3. MSPT 最大值是否远高于平均值，是否存在尖刺。
4. CPU 是否长期高占用。
5. 内存使用率是否过高。
6. GC 是否频繁或耗时异常。
7. 是否是持续卡顿，还是偶发尖刺。
8. 如果缺少 TPS/MSPT/GC 数据，请明确说明无法判断对应问题。

注意：

* TPS 像服务器心跳，正常一般接近 20。
* MSPT 是每次心跳花费时间，接近或超过 50ms 时容易影响 TPS。
* 不要在缺少 TPS/MSPT 的情况下断言玩家一定卡顿。
* 不要在缺少 GC 数据的情况下断言一定是 GC 问题。

## 4. 线程分析数据

{{threadData}}

请重点分析：

1. 主线程是否存在明显热点。
2. 主线程是否可能被同步 IO、数据库、区块加载、世界保存、实体 AI、红石/漏斗/方块更新阻塞。
3. 异步线程是否只是后台任务，不能直接等同于卡顿。
4. wait / sleep / park / idle / blocking queue 等状态是否可能只是线程空闲或等待任务。
5. 是否有插件、模组、类名、包名、线程名反复出现。
6. 热点是集中在少数代码，还是分散在大量任务中。
7. 如果缺少主线程堆栈，请不要断言具体插件导致主线程卡顿。

## 5. 来源分析数据

{{sourceData}}

请重点分析：

1. 热点来源属于 Minecraft 原版、插件、模组、Java/JVM、数据库、网络、未知来源中的哪类。
2. 是否有某个插件或模组占比明显偏高。
3. 是否有多个来源互相印证同一个问题。
4. 是否只是在异步线程中出现，还是影响主线程。
5. 如果来源识别不完整，请保守判断。

注意：

* 只要某个插件/模组出现在报告中，不代表它一定有 bug。
* 只能说“该来源在报告中占比较高”或“可能相关”。
* 只有在多个证据互相印证时，才能提高置信度。

## 8. 诊断要求

请按以下顺序思考，但最终只输出 JSON：

1. 先判断报告类型和数据完整性。
2. 判断是否存在真实性能问题。
3. 判断问题是持续性的，还是偶发尖刺。
4. 判断是否与主线程有关。
5. 判断是否与内存 / GC 有关。
6. 判断是否与插件、模组、实体、区块、红石、数据库 IO、世界保存有关。
7. 给出小白能理解的解释。
8. 给出专业分析和证据。
9. 给出安全、可执行、可回滚、可复测的修复计划。
10. 给出下一次应该如何重新采集 spark 报告。

---

## 9. 修复建议要求

每条修复建议都必须包含：

1. 小白步骤：普通服主能照着做。
2. 专业说明：解释背后的性能原理。
3. 风险说明：这项操作可能带来什么副作用。
4. 回滚方式：改坏了怎么恢复。
5. 复测方法：如何用 spark、TPS、MSPT、玩家反馈或日志确认是否有效。
6. 证据来源：这条建议为什么和当前报告相关。

如果建议涉及配置修改、插件/模组调整、JVM 参数、数据库设置，必须提醒：

* 先备份。
* 一次只改一项。
* 改完后重新采样对比。
* 不要在高峰期直接大改生产服。
