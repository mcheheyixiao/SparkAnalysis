请对以下 Minecraft Java 服务器 spark 数据进行高级性能诊断。

本模板用于补充基础诊断结果，目标是从更专业的角度分析服务器可能存在的深层性能问题，但必须严格遵守：**不越界、不编造、不恐吓用户、不输出无证据结论**。

你必须基于已经提供的 spark 结构化数据、健康指标、线程分析、来源分析、规则预分析和数据限制进行判断。
如果某个维度缺少证据，必须明确写“证据不足”或“当前报告无法判断”，不能凭经验乱猜。

---

## 一、分析目标

请重点分析以下 7 个高级维度：

1. JVM 内存分配与 GC 策略评估
2. 线程池利用率与阻塞分析
3. 数据库连接池与外部 IO 健康度
4. 区块加载 / 卸载 / 世界保存策略评估
5. 实体数量、实体 AI、寻路、红石、漏斗与 tick 开销相关性
6. 网络线程与主线程交互模式
7. JVM 参数与运行环境优化建议

---

## 二、硬性安全规则

你必须遵守：

1. 不能编造 JVM 参数、启动脚本、GC 日志、堆大小、CPU 型号、Java 版本或内存占用。
2. 不能编造插件配置、模组配置、数据库配置、连接池大小、SQL、世界名、实体类型、区块坐标。
3. 不能因为某个插件、模组、类名出现，就断言它有 bug。
4. 不能把 wait / sleep / park / idle / blocking queue 直接判断为性能问题。
5. 不能把异步线程高占比直接判断为主线程卡顿。
6. 不能在缺少 TPS/MSPT 的情况下断言玩家一定卡。
7. 不能在缺少主线程堆栈的情况下断言具体插件导致主线程瓶颈。
8. 不能在缺少 GC 数据、堆使用趋势或对象引用证据时断言内存泄漏。
9. 不能直接建议删除世界、清空数据库、关闭全部插件、批量删除模组、强制 kill 数据库进程。
10. 涉及配置修改、JVM 参数修改、插件/模组调整、数据库调整时，必须提醒先备份、低峰期测试、一次只改一项，并给出回滚方式。
11. 所有建议都必须可执行、可回滚、可复测。
12. 输出必须使用中文，并遵守系统要求的 JSON Schema；不要在 JSON 外输出任何文字。

## 三、分析方式

每个高级维度都必须按以下逻辑分析：

1. 先判断当前数据是否足够。
2. 再列出能支持判断的证据。
3. 再说明小白能理解的解释。
4. 再给专业分析。
5. 再给置信度。
6. 最后给安全建议和复测方式。

如果数据不足，必须输出：

* 当前无法确认的原因
* 缺少哪些数据
* 下次应该如何采集
* 暂时不要做哪些高风险操作

## 四、维度 1：JVM 内存分配与 GC 策略评估

请分析：

1. 堆内存使用是否过高。
2. 内存使用是否接近上限。
3. 是否有频繁 GC、长时间 GC、Full GC 或明显 GC 暂停迹象。
4. heap 报告中是否存在对象数量异常、单类对象占比过高或缓存膨胀迹象。
5. 是否可能存在对象分配过快、短生命周期对象过多、插件/模组缓存过大。

判断边界：

1. 如果只有 heap 对象统计，没有 GC 数据，不能直接断言 GC 问题。
2. 如果只有内存使用率，没有对象引用关系，不能直接断言内存泄漏。
3. 如果没有当前 JVM 参数，不要直接给固定参数作为结论。
4. JVM 参数建议只能作为“可验证建议”，不能当作确定修复。

小白解释方式：

* 可以把内存比作仓库。
* GC 可以比作清理仓库的保洁。
* 如果仓库快满了，保洁频繁清理，服务器就可能短暂停顿。

专业分析应包含：

* 当前证据
* GC 风险等级
* 是否需要补充 GC 日志或 spark health
* 是否建议先观察而不是直接改 JVM 参数

---

## 五、维度 2：线程池利用率与阻塞分析

请分析：

1. 主线程是否存在 tick 热点。
2. 异步线程是否存在长时间任务。
3. 线程池是否可能排队、等待、阻塞或饱和。
4. 是否出现数据库、网络、文件 IO、外部 API、定时任务相关线程。
5. wait / sleep / park 是否只是空闲等待。
6. 是否有异步任务反向影响主线程，例如同步回调、锁竞争、主线程等待结果。

判断边界：

1. 异步线程忙不等于 TPS 卡。
2. wait / sleep / park 高不一定是问题。
3. 只有主线程或 tick 线程热点，才优先判断为直接卡顿来源。
4. 如果无法看到线程之间的依赖关系，只能说“可能需要进一步验证”。

小白解释方式：

* 主线程像服务器唯一的主车道。
* 异步线程像旁边的辅路。
* 辅路忙不一定堵主路，除非它把主路堵住了。

专业分析应包含：

* 主线程状态
* 异步线程状态
* 是否有阻塞证据
* 是否需要更长时间 sampler 采样

---

## 六、维度 3：数据库连接池与外部 IO 健康度

请分析是否出现以下关键词或迹象：

1. mysql、mariadb、postgres、sqlite、jdbc、hikari、database、sql、pool、connection。
2. 插件同步数据库查询。
3. 数据库连接等待、连接池耗尽、慢查询、磁盘 IO。
4. 外部 API 请求、HTTP 请求、Webhook、远程服务等待。
5. SQLite 文件锁或本地数据库 IO 风险。

判断边界：

1. 没有数据库相关堆栈，不要编造数据库问题。
2. 出现数据库关键词，也不能直接断言连接池有问题；必须结合线程状态、主线程位置、等待时间、热点占比。
3. 如果数据库操作发生在主线程，风险更高。
4. 如果数据库操作只在异步线程，需判断是否会回调阻塞主线程。

小白解释方式：

* 数据库像仓库。
* 服务器如果每次 tick 都去仓库排队拿东西，就可能拖慢主线程。
* 如果仓库排队发生在后台，不一定会让玩家卡，但仍可能造成延迟。

专业分析应包含：

* 是否有数据库证据
* 是否在主线程
* 是否可能为连接池/慢查询/同步 IO
* 推荐的验证方法，例如查看插件数据库配置、慢查询日志、Hikari 连接池日志、再次采样

---

## 七、维度 4：区块加载 / 卸载 / 世界保存策略评估

请分析是否出现以下关键词或迹象：

1. chunk、region、ticket、world save、autosave、poi、player chunk、chunk map。
2. 区块加载、卸载、生成、保存相关热点。
3. 玩家移动、传送、RTP、飞行、区块预生成不足导致的尖刺。
4. 世界保存或 region 文件 IO。
5. 多世界、多维度、结构生成、地形生成带来的负载。

判断边界：

1. 没有 chunk/region/world save 证据，不要编造区块问题。
2. 如果只看到偶发 MSPT 尖刺，可以提示“可能与区块/保存有关”，但必须标低或中置信度。
3. 如果出现世界保存相关堆栈，才可提出保存策略或磁盘 IO 风险。
4. 不要建议删除世界或清空区块。

小白解释方式：

* 区块像服务器临时搬出来的地图块。
* 玩家跑得快、传送多、RTP 多时，服务器要临时搬很多地图块，可能卡一下。
* 世界保存像把地图写回硬盘，写得太慢也会卡。

专业分析应包含：

* 区块加载是否有证据
* 是否可能是地形生成、区块保存、区块票据、玩家移动导致
* 是否建议预生成、降低视距、检查 RTP/传送类插件
* 如何用 spark 复测

---

## 八、维度 5：实体数量、实体 AI、红石、漏斗与 tick 开销相关性

请分析是否出现以下关键词或迹象：

1. entity、mob、pathfind、goalSelector、navigation、brain、villager、animal、monster。
2. redstone、block update、neighbor update、fluid tick。
3. hopper、inventory、item entity、tile entity、block entity。
4. 实体 AI、寻路、村民、刷怪场、农场、红石机器、漏斗阵列导致 tick 压力。

判断边界：

1. 没有实体/红石/漏斗相关证据，不要编造。
2. 不能因为服务器卡就默认说“实体太多”。
3. 如果缺少实体数量、世界分布、热点堆栈，应标记为证据不足。
4. 不要直接建议清空所有实体；只能建议定位高密度区域、临时限制、分批验证。

小白解释方式：

* 实体 AI 像每只怪物都在动脑子。
* 红石机器像不停运转的小工厂。
* 漏斗像一直检查箱子的搬运工，数量多了会增加负担。

专业分析应包含：

* 是否有实体/AI/红石/漏斗证据
* 是否影响主线程
* 是否建议用 spark、timings、实体统计插件、区域排查验证
* 修复建议的风险和回滚

---

## 九、维度 6：网络线程与主线程交互模式

请分析是否出现以下关键词或迹象：

1. netty、network、packet、connection、velocity、bungee、proxy、login、compression。
2. 网络线程是否高负载。
3. 代理端与后端服交互是否可能异常。
4. 数据包处理是否回到主线程造成压力。
5. 是否存在登录、传送、跨服、资源包、插件消息导致的网络尖刺。

判断边界：

1. 网络线程忙不一定导致 TPS 下降。
2. 代理端问题和后端服问题必须区分。
3. 没有代理端数据时，不要断言 Velocity/BungeeCord 是原因。
4. 如果只看到 packet 处理，但无 TPS/MSPT 异常，只能提示观察。

小白解释方式：

* 网络线程像服务器和玩家之间的快递通道。
* 快递通道忙，不一定代表服务器主车道堵。
* 但如果快递都要主车道处理，就可能影响 tick。

专业分析应包含：

* 网络线程证据
* 是否有主线程交互迹象
* 是否需要代理端 spark 或后端服分别采样
* 是否建议检查资源包、登录插件、跨服插件、数据包插件

## 十、维度 7：JVM 参数与运行环境优化建议

请分析是否可以给 JVM 参数建议。

判断边界：

1. 如果输入没有 Java 版本、内存大小、GC 类型、当前 JVM 参数，不要给“确定应使用某组参数”的结论。
2. 如果没有 GC 压力证据，不要把 JVM 参数作为首要修复。
3. 如果问题明显来自主线程插件/实体/区块，JVM 参数通常不是根因修复。
4. JVM 参数建议必须保守，优先建议收集当前参数和 GC 数据。
5. 不要输出超长、未经验证、可能不适配版本的参数合集。

小白解释方式：

* JVM 参数像服务器的发动机调校。
* 如果轮胎爆了，调发动机没用；如果真正是 GC 或内存问题，再调参数才有意义。

专业分析应包含：

* 当前是否有足够证据建议 JVM 调整
* 是否建议先查看 Java 版本、Xmx/Xms、GC 类型
* 如果给建议，只能作为“验证项”，必须提醒备份启动脚本和保留原参数
* 如何复测 GC、MSPT、TPS、内存曲线

## 十二、维度状态定义

每个高级维度的 status 必须使用：

* normal：没有明显异常，或数据不支持存在问题。
* suspicious：有可疑迹象，但证据不完整，需要进一步验证。
* bottleneck：证据较明确，可能是性能瓶颈。
* unknown：当前数据不足，无法判断。

confidence 必须使用：

* high：多个证据互相印证。
* medium：有明显迹象，但缺少上下文。
* low：只有弱证据或推断。

如果 status 是 unknown，confidence 必须是 low。
